## 1. 깨끗한 코드

#### 나쁜 코드

프로그래머라면 누구나 나쁜 코드를 수 없이 경험했을 것이다. 그렇다면, 왜 나쁜 코드를 짰는가?

> "프로젝트 데드라인이 너무 임박해서 어쩔 수 없었어요"
>
> "코드를 리팩토링하기엔 시간이 없었어요? 제 입장을 아시나요?!"
>
> "리팩토링을 하고싶었지만 상사가 그걸 봤다면 분명 놀고있다고 했을거에요"
>
> "같은 코드만 보자니 너무 지겨워서요, 동작만 되면 되는거 아닌가요? 얼른 끝내고 다른 업무하려했죠"



> "뭐, 안 돌아가는 프로그램보단, 돌아가는 쓰레기가 낫죠. 일단 돌아가면 나중에 리팩토링 할려고 했다고요~~"



가지각색의 핑계들 중 정말 여러분들의 핑계는 없을까요???

나중에 하려던 코드 리팩토링은 정말로 하셨나요???

뭐, 괜찮습니다. 더 좋은 프로그래머가 되기 위해 자는 시간, 노는 시간 재껴두고 지금 이걸 읽고 있는 거잖아요?



#### 나쁜 코드의 대가

여러분들은 유지 보수를 하기 위해 남이 짜놓은 코드를 읽습니다.

그리고 생각나는 몇 마디들,

> "이 코드 짠 새X 내 앞에 데려와" or
>
> "어떤 개XX가 이런 커다란 똥을 싸던지고 튄거야"

여러분들도 이해될 것이고, 저 역시 이해가됩니다.

당장 느려도됩니다. 시간이 지날 수록 조금이라도 상승 곡선을 보여주면됩니다.

가장 좋은 방법은 처음부터 번개같은 속도로 개발을 하고, 시간이 지나면서 음속의 개발 속도를 보여주면됩니다!

할 수 있다면 말이죠. 하지만, 대부분은 초반에 번개 같은 속도에 후반은 지쳐 굼뱅이 같은 속도의 생산성을 보일테죠.

번개 같은 속도로 개발하기 위해 얽히고설킨 코드를 '해석'하고 얽히고 설킨 코드를 추가하고 나중에 또 이 코드를 '해석'하죠.

그리고, 조그만 시간이 지나면 이 얽히고설킨 코드가 앞서 말한 '커다란 똥'이 됩니다.



이것을 없에기 위해 인력을 추가하지만 역부족입니다. 애초에 '인력 문제'가 아니기 때문이죠.

그리고, 말합니다.

> "이 짓거리 못 하겠어! 아무래도 재설계를 해야할 것 같아!!"

자, 희망 가득한 새로운 팀이 만들어졌습니다. 그리고, 새로운 팀은 "기존 시스템의 기능 + 클린 코드"를 통해 생산성을 회복해야합니다! 이것은 절호의 기회입니다.



새로운 팀이 기존 시스템을 따라잡을 쯤, 새로운 팀에 새로운 인력이 추가되고 그들은 말합니다.

> "이 짓거리 못 하겠어! 아무래도 재설계를 해야할 것 같아!!"



이러한 악순한의 연결 고리를 벗으려면 결국 최대한 시간을 투자해 깨끗한 코드를 짜야합니다.

깨끗한 코드는 이런 비용들을 절감시키고 프로그래머로서의 가치를 높여줄겁니다.



#### 태도

지나가다 팀원의 코드를 봤다. 이럴수가… 입을 다물 수 없었다.

코드가 너무 엉망이였던 것이다. 왜 그런지 물었더니 대뜸 정색을 하고 말한다.

> "기존 코드는 완~전 잘 짰는데, 갑자기 요구 사항을 바꿔서 코드를 뒤집어 엎었다고!!"
>
> "기간을 여유없이 줬는데, 제대로 짤 시간이 있겠어?? 데드 라인이 코 앞인데?!"
>
> "능력없는 기획자랑 말하면 뚝딱 뚝딱 되는 줄 아는 성질급한 고객과 쓸데없는걸 요구하는 마케팅 부서 때문이야!"



인용구를 읽고 바보처럼 웃고 있는 당신....! 알잖아요. 본질적인 문제는 결국 프로그래머 잘못이란걸...

코딩을 하고있는 당신과 나쁜 코드는 결국 프로그래머의 잘못이 크다는 걸 얘기할 수 밖에 없습니다.

> "그러면 위에서 까라는데 개겨요?"



일정과 요구 사항이 그들의 '책임'인 것이고 좋은 코드는 '프로그래머의 책임'이다.



식당과 손님이 있다고하자. 손님이 갑이고 식당이 을이다.

식당에 손님이 배가 너무 고파서 고기를 날로 먹을려고한다. 그러면 식당 주인은 손님이 갑이기 때문에 그냥두나?? 아니다.

분명 "손님 날로 먹으면 분명 배탈이 날거에요!"라고 말릴 것이다. 식당 주인은 '전문가'니까!

프로그래밍 전문가는 당신이다. 나쁜 코드를 날로 먹겠다는데 후에 배탈이 나든 말든 가만히 두는건 '전문가'가 할 일은 아니지않나?



#### 원초적 난제

솔직하게, 시간 문제상 나쁜 코드를 짤 수 밖에 없다는건 정말 '핑계'다.

나쁜 코드를 짜면 위에 글들 처럼 결국 생산성은 떨어지고 기한을 맞추지 못할 것이다.

깨끗한 코드를 짜고 유지하는 것은 '습관'이고, 그 습관들은 당신을 주어진 기간에 맞춰주고 칼퇴근을 할 수 있게해줄 것이다.



#### 깨끗한 코드라는 예술?

위에 말들을 모두 납득했다. 좋다!

그래서? 깨끗한 코드는 어떻게 짜는 것인가. 깨끗한 코드는 하나의 예술 작품이다.

어떠한 코드를 보고 정말 아름답다고 생각한 적이 있는가? 필자는 있다.

인턴기간에 어떤 분이 짠 코드를 CTO가 보고 감탄을 하고 있길래 가서 봤다. 차마 말을 잇지 못했다. 코드가 저렇게 예쁠 수 있구나란걸 처음으로 느꼈다.

그렇다. 나는 깨끗한 코드가 뭔지 눈으로 보고 알 수 있었다. 그런데? 깨끗한 코드가 뭔지 알고 있는 나는 깨끗한 코드를 짤 수 있는가? 정답은 당연히 NO다.

미식가라고 음식을 쉐프급으로 잘 할 수있지는 않지 아니한가.

깨끗한 코드를 짤 수 있는 사람은 가지고 있는 '코드 감각'과 힘들게 공부해서 얻은 여러 '코드 기법'을 적용하는 '절제와 규율'이 필요하다. 코드 감각이 있는 개발자는 나쁜 코드를 볼 수는 있어도 그걸로 끝이다. 반면, 코드 감각이 있는 개발자는 나쁜 코드를 좋은 코드로 개선할 방법을 생각하고 실천 계획을 한다.

#### 깨끗한 코드란?

##### 비야네 스트롭스트룹 (C++ 창시자)

> 나는 우아하고 효율적인 코드를 좋아한다. 논리가 간단해야 버그가 숨어들지 못한다. 의존성을 최대한 줄여야 유지보수가 쉬워진다. 오류는 명백한 전략에 의거해 철저히 처리한다. 성능을 최적으로 유지해야 사람들이 원칙 없는 최적화로 코드를 망치려는 유혹에 빠지지 않는다. 깨끗한 코드는 한 가지를 제대로 한다.

우아한 코드란 즉, '보기 좋은 코드'다.

나쁜 코드는 나쁜 코드를 '유혹'한다. 실용주의 프로그래머 데이브 토마스와 앤디 헌트는 같은 이야기를 깨진 창문으로 비유를 했다. 창문이 깨진 건문은 누구도 상관하지 않기 때문에, 그 건물의 남은 창문들이 더 깨져도 역시 개의치 않는다. 그러다보면, 누구는 자발적으로 창문을 깨고, 창문 뿐만아니라 건물의벽에 낙서를하고 쓰레기를 함부로 버린다.

한 가지를 제대로 한다고 한다. 나쁜 코드는 너무 목적이 많아 이것도하고 저것도 하려다 의도가 뒤섞여 결국 목적들이 흐려진다.



##### 그래디 부치

> 깨끗한 코드는 단순하고 직접적이다. 깨끗한 코드는 잘 쓴 문장처럼 읽힌다. 깨끗한 코드는 결코 설계자의 의도를 숨기지 않는다. 오히려 명쾌한 추상화와 단순한 제어문으로 가득하다.

그래디는 '가독성'을 중요시하며, 코드는 사실레 기반해야 함을 담고있다.



##### 큰 데이브 토마스

> 깨끗한 코드는 작성자가 아닌 사람도 읽기 쉽고 고치기 쉽다. 단위 테스트 케이스와 인수 테스트 케이스가 존재한다. 깨끗한 코드에는 의미 있는 이름이 붙는다. 특정 목적을 달성하는 방법은 (여러 가지가 아니라) 하나만 제공한다. 의존성은 최소이며 각 의존성을 명확히 정의한다. API는 명확하며 최소로 줄였다. 언어에 따라 필요한 모든 정보를 코드만으로 명확히 표현할 수 없기에 코드는 문학적으로 표현해야 마땅하다.

남들이 고치기 쉬운 코드여야하며, 아무리 우아하고 가독성이 좋은 코드여도 테스트 케이스가 없는 코드는 깨끗한 코드가 아니다.



##### 마이클 페더스

> 깨끗한 코드의 특징은 많지만 그 중에서도 모두를 아우르는 특징이 하나 있다. 깨끗한 코드는 언제나 누군가 주의 깊게 짰다는 느낌을 준다. 고치려고 살펴봐도 딱히 손 댈 곳이없다. 작성자가 이미 모든 사항을 고려했으므로. 고칠 궁리를 하다보면 언제나 제자리로 돌아온다. 그리고는 누군가 남겨준 코드, 누군가 주의 깊게 짜놓은 작품에 감사를 느낀다.

깨끗한 코드는 세세한 부분까지 주의 깊게 작성한 코드다.



##### 론 제프리스

> 최근 들어 나는 켄트 백이 제안한 단순한 코드 규칙으로 수현을 시작한다. (그리고 같은 규칙으로 구현을 거의 끝낸다.) 중요한 순으로 나열하자면 간단한 코드는
>
> - 모든 테스트를 통과한다
> - 중복이 없다
> - 시스템 내 모든 설계 아이디어를 표현한다
> - 클래스, 메서드, 함수 등을 최대한 줄인다
>
> 물론 나는 주로 중복에 집중한다. 같은 작업을 여러 차례 반복한다면 코드가 아이디어를 제대로 표현하지는 못한다는 증거다. 나는 문제의 아이디어를 찾아내 좀 더 명확하게 표현하려 애쓴다.
>
> 내게 있어 표현력은 의미 있는 이름을 포함한다. 보통 나는 확정하기 전에 이름을 여러 차례 바꾼다. 이클립스와 같은 최신 개발 도구는 이름을 바꾸기가 상당히 쉽다. 그래서 별 고충 없이 이름을 바꾼다. 하지만 표현력은 이름에만 국한되지 않는다. 나는 여러 기능을 수행하는 객체나 메서드도 찾는다. 객체가 여러 기능을 수행한다면 여러 객체로 나눈다. 메서드가 여러 기능을 수행한다면 메서드 추출 리팩터링 기법을 적용해 기능을 명확히 기술하는 메서드 하나와 기능을 실제로 수행하는 메서드 여러 개로 나눈다.
>
> 중복과 표현력만 신경 써도 (내가 생각하는) 깨끗한 코드라는 목표에 성큼 다가선다. 지저분한 코드를 손볼 때 이 두 가지만 고려해도 코드가 크게 나아진다. 하지만 나는 한 가지를 더 고려한다. 이는 설명하기 조금 까다롭다.
>
> 오랜 경험 끝에 나는 모든 프로그램이 아주 유사한 요소로 이워진다는 사실을 깨달았다. 한 가지 예가 '집합에서 항목 찾기'다. 직원 정보가 저장돤 데이터 베이스든, 키/값 쌍이 지정된 해시 맵이든, 여러 값을 모아놓은 배열이든, 프로르맹을 짜다 보면 어떤 집합에서 특정 항목을 찾아낼 필요가 자주 생긴다. 이런 상황이 발생하면 나는 추상 메서드나 추상 클래스를 만들어 실제 구현을 감싼다. 그러면 여러 장점이 생긴다.
>
> 이제 실제 기능은 아주 간단한 방식으로 예를 들어 해시 맵으로, 구현해도 괜찮다. 다른 코드는 추상 클래스나 추상 메서드가 제공하는 기능을 사용하므로 실제 구현은 언제든지 바꿔도 괜찮다. 지금은 간단하게 재빨리 구현했다가 나중에 필요할 때 바꾸면 된다.
>
> 게다가 집합을 추상화하면 '진짜' 문제에 신경 쓸 여유가 생긴다. 간단한 찾기 기능이 필요한데 온갖 집합 기능을 구현하느라 신간과 노력을 낭비할 필요가 없어진다.
>
> 중복 줄이기, 표현력 높이기, 초반부터 간단한 추상화 고려하기. 내게는 이 세 가지가 깨끗한 코드를 만드는 비결이다.

중복을 피하고, 한 가지 기능만 제대로 수행하며, 제대로 표현하고 작게 추상화해라.



##### 워드 커닝햄

> 코드를 읽으면서 짐작했던 기능을 각 루틴이 그대로 수행한다면 깨끗한 코드라 불러도 되겠다. 코드가 그 문제를 풀기 위한 언어처럼 보인다면 아름다운 코드라 불러도 되겠다.

코드를 읽었을 때 '이렇게 동작하겠네'라고 생각을 하면 그렇게 동작이 되고, 해당 코드가 목적에 맞게 문제를 풀어야 한다.



##### 로버트 C. 마틴 (이 책 저자)

깨끗한 변수 이름, 깨끗한 함수, 깨끗한 클래스를 만들어야한다. 코드를 읽는 시간은 코드를 짜는 시간에 10배 정도는 길다. 새 코드를 짜면서 우리는 끊임없이 기존 코드를 읽는다.

읽기 쉬운 코드, 깨끗한 코드가 어째서 매우 중요한지 느낌이 올 것이다. 읽기 쉬운 코드를 짜는 일은 쉽지 않지만, 기존 코드를 읽어야 새 코드를 짜기 쉬워진다.

그러므로 급하고, 서둘러 끝내야 하고, 쉽게짜려면 읽기 쉽게 만들면된다.



#### 보이스카우트 규칙

잘 짠 코드는 시간 대비 깨끗함이 유지되어야 한다. 시간이 지나면서 엉망이 된 코드가 한 둘이 아니다.

> 캠프장은 처음 왔을 때보다 더 깨끗한게 해놓고 떠나라.

미국 보이스카우트의 간단한 규칙은 우리에게도 유용하다.

한꺼번에 많은 시간을 들여 코드를 정리하라는 것이 아니다. 위에서 말한대로 '습관'을 들이면된다.

##### 변수 이름을 개선하고, 복합적인 기능의 함수는 분할하고, 중복은 제거하고, 복잡하고 네스팅된 if 문을 정리하면 충분하다.
